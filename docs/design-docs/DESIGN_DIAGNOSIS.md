# CS Cup Manager — 设计诊断报告

> **作者**: GameDesigner
> **日期**: 2026-06-26
> **版本**: v1.0
> **范围**: 完整设计审查 — 从 GDD 到代码实现的一致性、完整性和平衡性分析

---

## 一、总体评估

当前设计文档层面的系统骨架是**扎实且自洽的**——策略矩阵、特质系统、化学反应规则、条件分层、侦察记忆等机制都有明确的触发条件和效果描述。问题不在"想了什么"，而在于**代码和文档之间有一条巨大的鸿沟**。

核心症结：`src/simulation/cup.ts` 的 `resolveMatch` 函数本质上是一个**加权掷骰子**，GDD 中描述的几乎所有机制都未进入模拟层。

## 二、缺陷清单（按严重度排序）

### 🔴 P0 — 核心循环断裂

| # | 缺陷 | 当前状态 | 玩家体验后果 |
|---|---|---|---|
| **1** | 比赛结果 ≈ 随机 | `teamPower()` 加权 + 随机 `(-9, 9)`，无策略碰撞 | 玩家的战术选择对胜负几乎没有影响，挫败感极强 |
| **2** | 策略矩阵未接入 | GDD 定义了 4×4 攻防碰撞表（16 种组合），代码中只有硬编码 `"全起 + 默认架枪"` | 所有对手"打法"相同，没有博弈感 |
| **3** | 玩家决策触点为 1 | 9 场比赛中每场只有一次硬编码决策（开局），GDD 要求 2-3 次/场 | 经理游戏变成"按按钮看骰子" |
| **4** | 经济系统为假系统 | 无回合间金钱追踪、"全起"永远可用、无 eco/save/force-buy 实际后果 | 经济维度完全消失，GDD 中"经济与战术分开"的要求未实现 |

**诊断**：这四个缺陷合在一起意味着——当前版本的"比赛"不是一个有决策深度的系统，而是一个**按下按钮后看随机结果的过场动画**。这是最危险的：一旦玩家意识到自己的选择没有意义，整个游戏就崩溃了。

### 🟠 P1 — 关键系统缺失

| # | 缺陷 | 当前状态 | 风险 |
|---|---|---|---|
| **5** | 特质效果未计算 | 特质字段存在于 content，GDD 定义了 ±3-8 的修正，但 match resolver 完全不读 traits | 选人变成纯数值比较，没有个性维度 |
| **6** | 化学反应规则未生效 | 10 条规则（星+稳、双核情绪、孤核等）全在文档中，代码只用 `cohesion * 0.18` 加权 | 组队策略扁平化——买最贵的五个人就完事了 |
| **7** | 凝聚力阈值事件不存在 | <30 解锁负面链、>80 解锁正面链——定义了但无触发代码 | 凝聚力在 30-80 之间只是一个没有意义的数字 |
| **8** | 暂停系统未实现 | 每场 1 次暂停 + 三种选择（战术/情绪/纪律重置），代码中零实现 | 玩家失去了"关键时刻扭转局势"的操作空间 |
| **9** | 侦察系统未接入 | 赛前侦察卡定义完好但无生成逻辑，对手倾向追踪为空 | 没有赛前准备维度，比赛永远是"盲打" |
| **10** | 事件系统完全断开 | EVENTS.md 和 MECHANICS.md 定义了丰富的事件库，`resolveMatch` 不调用任何事件 | 每场比赛读起来都一样，没有叙事差异 |

### 🟡 P2 — 平衡性与长期风险

| # | 缺陷 | 风险描述 |
|---|---|---|
| **11** | **雪球效应**：赢 → 有钱 → 买好选手 → 继续赢 | 无逆转机制。第一个杯赛输了基本等于整个赛季报废。经济文档承认此问题但未解决 |
| **12** | 教练哲学无实现 | 4 种风格的差异只存在于文档，无 UI 选择、无模拟效果 |
| **13** | 转会系统为纯文本占位 | 出价逻辑、球员意愿、角色承诺——全在文档中未实现。赛季间无管理操作 |
| **14** | 替补系统无意义 | 第 6 人只用于"缺席事件"，但缺席事件不存在。替补是纯粹的预算负担 |
| **15** | 赛后反馈无透明度 | GDD 设计了详细的决策反馈格式（`战术执行 84 vs 77，装备劣势 -5`），当前代码只输出硬编码文本 |

### 🔵 P3 — 设计文档本身的问题

| # | 问题 | 说明 |
|---|---|---|
| **16** | 魔法数字过多且无调优入口 | 权重系数（0.38/0.28/0.18/0.16）没有"为什么是这些值"的解释，没有外部化配置 |
| **17** | 随机范围过大 | `±9` 在一个 ~50-90 的 power scale 上意味着 ±10-18% 的摇摆——这远远超过策略差异的影响，会淹没有意义的决策 |
| **18** | 奖金数值全部缺失 | ECONOMY.md 大量 `[PLACEHOLDER]` 未填，无法做经济平衡模拟 |

## 三、系统间依赖断裂图

以下展示 GDD 承诺的机制链 vs 当前实现的实际情况：

```
GDD 承诺的数据流：
  特质 → 化学反应 → 凝聚力 → 事件触发 → 决策修正 → 策略碰撞 → 结果生成
  经济 → 装备质量 → 火力修正 ──────────────┘
  侦察 → 策略预读 → 碰撞加成 ──────────────┘

当前代码的实际流：
  teamPower(加权四维) + matchPointBonus(±3) + random(±9) → 比较大小 → 谁大谁赢
```

**结论**：所有横向箭头在代码中都不存在。玩家体验到的只有一个维度——"我的队伍 power 够不够高"。

## 四、三个核心设计问题

### 问题 1："经理"的操作空间在哪里？

当前设计把经理定义为"做战术决策的人"，但如果你只能在一场 5 局比赛里做 1 次选择，这就不是经理游戏，是**抽卡 + 观战**。

**需要回答的设计问题**：
- 每场比赛 2-3 次决策确实够了，但决策的**权重**够大吗？
- 如果玩家做了完美决策但队伍实力差距 20%，能翻吗？翻的概率是多少？
- 如果翻不了——那就不是"我的决策赢的"，是"我抽到强队赢的"。

**建议**：决策修正值应该能覆盖 60-70% 的纯实力差距。也就是说如果玩家通过侦察 + 克制策略 + 暂停调整做到了最优决策链，应该能击败实力高出 10-15 分的对手。

### 问题 2：比赛的"故事感"从哪里来？

GDD 非常强调叙事卡片和文本反馈，但叙事需要**状态驱动**。如果比赛没有追踪士气、没有追踪特质链、没有追踪教练缺席，那事件卡片就只能是随机模板拼接。

**建议**：比赛引擎需要维护一个轻量状态机，最少包含：
- 当前经济等级（5 档：全起/半起/强起/eco/破产）
- 当前士气状态（高涨/正常/低落/崩溃）
- 当前战术控制（正常/混乱/自主决策/教练缺席）
- 活跃特质链（当前正在触发的特质事件）

每一个状态变化都**自动生成一个事件卡片**。这样叙事就不是"编出来的"，而是"模拟出来的"。

### 问题 3：雪球怎么打破？

这是一个设计哲学问题。有三种主流方案：

| 方案 | 机制 | 优点 | 缺点 |
|---|---|---|---|
| A. 实力收敛 | 强队自动被削弱（"热门压力"），弱队自动被加强（"underdog buff"） | 简单 | 玩家会觉得不公平 |
| B. 事件逆转 | 落后方获得更有利的事件选择权 | 叙事合理 | 需要大量事件内容 |
| C. 资源重分配 | 杯赛奖金差距小，转会窗口给弱队优先权 | 经济驱动 | 实现复杂 |

**建议**：v0.1 采用方案 B + C 的组合。方案 A 会让玩家感觉被系统"暗算"，在单机游戏中体验很差。

## 五、改进路线图

### Phase 1 — 让一场比赛"活起来"（v0.1 必须完成）

这是**最小可行比赛引擎**。目标：玩家打一场比赛，能感受到决策有意义。

```
┌──────────────────────────────────────────────────────────┐
│ 回合 1: 开局部署                                          │
│   → 经济选择 (全起/半起/eco) → 装备质量修改器              │
│   → 战术选择 (rush/default/lurk/fake)                      │
│   → AI 基于侦察记忆选择防守策略                            │
│   → 碰撞结算 (策略矩阵 + 实力 + 特质 + 随机)               │
│   → 生成事件卡片 (含决策影响的反馈文本)                     │
├──────────────────────────────────────────────────────────┤
│ 回合 2-4: 压力/转折/调整                                  │
│   → 上一回合结果驱动经济状态变化                            │
│   → 士气/战术控制根据胜负更新                              │
│   → 特质链触发检测（hot_blooded 先冲 → 被孤立 → 减员）     │
│   → 暂停触发器检测（士气崩溃/特质链/对手连胜）             │
│   → 如果触发暂停：三选一决策 → 状态修正                     │
│   → 根据新状态再次战术决策                                 │
│   → 碰撞结算                                              │
├──────────────────────────────────────────────────────────┤
│ 回合 5: 终局                                               │
│   → 经济清算（最后的机会）                                 │
│   → 残局决策（拆包/保枪/强攻）                             │
│   → 最终结算 + 比赛 MVP + 关键转折点总结                    │
└──────────────────────────────────────────────────────────┘
```

**每场比赛最少 2 次决策，平均 3 次，最多 4 次。**

## 六、即时行动项（今天可做的数值调整）

这些是不需要大规模重构就能改善体验的微调：

### 6.1 缩减随机范围

```ts
// 当前：随机 ±9，约占 power 值的 10-18%
const aPower = teamPower(teamA) + (scoreA >= 2 ? 3 : 0) + rng.int(-9, 9);

// 建议：随机 ±4，让策略和实力成为主导
// 配合策略修正值（碰撞表 ±2~8）后，决策的权重才能被玩家感知
const aPower = teamPower(teamA) + (scoreA >= 2 ? 3 : 0) + rng.int(-4, 4);
```

### 6.2 给实力加权加上注释和外部化

```ts
// 当前权重是魔法数字
team.stats.firepower * 0.38 +
team.stats.tacticalExecution * 0.28 +
team.stats.cohesion * 0.18 +
team.stats.discipline * 0.16

// 建议：移到常量表并标注设计理由
// BALANCE_TABLE: firepower 决定击杀效率（最高权重，直接决定回合胜负）
// BALANCE_TABLE: tactics 影响策略碰撞成功率（次高，因为在 CS 中战术对胜负很重要）
// BALANCE_TABLE: cohesion 减缓负面事件影响 + 化学反应触发（中等权重）
// BALANCE_TABLE: discipline 减少失误和 eco 局劣势（中等权重，更偏向稳定性）
```

## 七、设计假设验证清单

以下假设需要在第一个可玩版本中验证。如果假设不成立，设计需要调整：

| # | 假设 | 验证方法 | 失败标准 |
|---|---|---|---|
| H1 | 每场 2-3 次决策能提供足够的管理参与感 | 内部试玩，问"你觉得你在指挥还是在看球？" | >50% 回答"在看球" |
| H2 | 策略矩阵的克制关系能被玩家感知 | 记录决策-胜率数据，看玩家是否会学习使用克制策略 | 克制策略胜率与非克制无显著差异 |
| H3 | 化学反应规则让选人有取舍而非最优解 | 统计不同化学组合的最终杯赛胜率 | 存在一种组合胜率 >70% |
| H4 | 随机因子（±4）不会让玩家觉得"运气决定一切" | 试玩后问卷 | >30% 反馈"输赢靠运气" |
| H5 | 20 分钟完成一个赛季仍然有满足感 | 计时并收集满意度 | 完成时间 <15 分钟且满意度 <6/10 |

## 八、总结：一句话版本

> **当前游戏是一个"按按钮看骰子"的抽卡模拟器，而不是一个经理决策游戏。核心问题不是设计文档写错了，而是设计中承诺的每一条机制都还没有进入代码。优先级最高的事：让一场比赛的 2-3 次决策真正改变比赛结果，让玩家感觉到"是我赢了"，而不是"我的队伍赢了"。**
