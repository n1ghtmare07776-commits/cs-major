# 真实性与合理性审计报告 — CS Cup Manager v0.1

> **审计人**: 游戏设计师
> **审计日期**: 2026-06-26
> **审计范围**: 全部源文件 (`src/`)、全部设计文档 (`docs/product-specs/`)、全部内容文件 (`src/content/`)
> **语气**: 刻薄但建设性。我不打算哄你。

---

## 一句话总结

**你的游戏给玩家开了透视挂、装了对手语音监听器、还让所有情报在比赛开始前就自动泄露到赛中的决策卡上。** 玩家不需要做任何侦察准备——因为游戏已经替他把对手的底裤都扒干净了。

---

## P0: 致命问题（如果上线，核心玩法直接报废）

### P0-1: 比赛中公开对手习惯 = 合法开挂

**问题**: 比赛决策卡中直接告诉玩家对手选手的个人习惯。

**证据**（来自截图反馈，与 `src/content/scout-briefings.ts` 的设计逻辑对照）:

```
比赛中决策卡显示:
"Team Spirit 第二时间的回防偏慢，但 sh1ro 会主动找首杀"
```

**为什么这是开挂**:

在真实的 CS 竞技中，你知道"sh1ro 喜欢主动找首杀"只有一个途径——赛前看了他一百场 demo，找了分析师做了热力图，统计了他的首杀率。这是**赛前情报工作**的成果，不是比赛进行中从天而降的提示。

你现在把这个信息直接贴在赛中的决策卡上，等于告诉玩家：

> "别担心，你不用看 demo、不用请分析师、不用做任何赛前准备——我们会帮你把对手的内裤颜色都标出来。"

这意味着 **`src/content/scout-briefings.ts` 里辛辛苦苦写的 827 行赛前情报系统变成了废纸**。因为玩家不需要在赛前做侦查选择——反正进了比赛什么都会知道。

**影响范围**:
- `src/content/scout-briefings.ts` — 整个赛前情报系统被架空
- `src/content/match-beats.ts` — 比赛中事件卡直接引用对手习惯
- `src/content/events.ts` — 事件系统未区分赛前/赛中信息层级
- `docs/product-specs/MECHANICS.md` 第 149-168 行 — Scouting Cards 的定义形同虚设

**修复预期**:
- 赛前情报 → 赛前侦察卡（`scout-briefings.ts`）**必须**是获取对手习惯的**唯一入口**
- 没有做侦察准备 → 比赛中**不能**出现对手具体习惯的提示
- 做了侦察准备 → 赛中事件的文本应该写成"根据侦察报告，你们预判了对手的前压习惯"，而不是"对手喜欢前压"

### P0-2: 比赛中监听对手语音 = 偷装窃听器

**问题**: 叙事文本直接引用对手队伍的语音通讯内容。

**证据**（来自 `src/content/match-beats.ts`）:

```
第 152 行: "{opponent}倒地的时候，{rival_team} 的语音里炸了：'首杀！B 点首杀！回防！'"
第 198 行: "{opponent} 的指挥在语音里吼：'他在哪？！他在哪？！'"
第 284 行: "他在语音里说：'我在他们后面。' 声音很平，像在说今天天气不错。"
```

加上 `src/content/scout-briefings.ts` 里也有:

```
第 172 行: "{rival_team} 的指挥在采访里说过一句话：'我不需要告诉 {key_player_name} 怎么打。我只需要确保他不缺子弹。'"
```

**为什么这是监听器**:

CS 的隔音棚是**物理隔音**的。选手戴的是**降噪耳机**。双方的语音通讯是**加密的局域网频道**。

你的玩家凭什么能听到对面队伍的语音？他是黑了 Discord 服务器还是把窃听器塞进了对方的鼠标垫里？

除非你的游戏设定里，每个教练都自带一个"读唇术+脑电波解析"的超能力——那我建议你把这个写在设计文档里，标题就叫《我们的游戏是玄幻题材》。

**影响范围**:
- `src/content/match-beats.ts` — 至少 8 处直接引用对手语音
- `src/content/scout-briefings.ts` — 引用对手选手私下言论
- 叙事基调 — 玩家被塑造成全知的上帝

**修复预期**:
- **全部删除**比赛中引用对手语音的叙事
- 己方语音可以保留——那是你自己队伍的通讯
- 对手的行为只能通过**外部可观察的表现**来描述（他们的站位、他们的枪线、他们的道具轨迹），不能通过他们说了什么
- 赛后采访可以引用对手言论——那是公开信息

### P0-3: 开场经济选项完全违背 CS 规则

**问题**: 游戏提供了第 1 回合"全买"的选项，但 CS 第一局永远是手枪局（$800）。

**证据**（来自 `src/content/mechanic-events.ts` 第 360-399 行）:

```typescript
// opening_economy_choice 事件
{
  id: "opening_economy_choice",
  roundType: ["opening"],
  title: "开场经济",
  narrative: "比赛开始。冻结时间。你的队长在语音里问：'教练，这把怎么买？'",
  choices: [
    {
      label: "全买：满甲满道具，全力以赴",
      resultText: "'全买。让他们看看我们开场是什么水平。'\n\n五个人满甲满道具..."
      effect: { firepower: 4, economy: -3, morale: 3, momentum: 2, tactical_control: 1 },
    },
    {
      label: "半买：步枪+甲，省点道具",
      // ...
    },
    {
      label: "ECO：手枪局，攒钱下把翻盘",
      // ...
    },
  ],
}
```

**为什么是灾难**:

任何看过哪怕一场 CS 比赛的人都知道：**第一局是手枪局。永远。** 双方 $800。你见过谁在 CS 比赛第一局起 AK 加满甲的？

这个问题在 `docs/design-docs/CS_MECHANICS_REFERENCE.md` 的第 3 章中已经详细说明了正确的经济流程，但 `mechanic-events.ts` 里的代码**完全没改**。这就是 GDD 和实现之间的经典断层——设计文档写对了，代码写错了，然后上线的是代码。

**修复预期**:
- `mechanic-events.ts` 的 `opening_economy_choice` 必须重写为**手枪局独有选项**
- 手枪局的可选范围：全甲+手枪 / 半甲+道具 / 沙鹰豪赌 / 省钱留装备 / CT 拆弹套件
- 不是"全买/半买/ECO"三选一——那是对第 3+ 局的选择

## P1: 严重问题（破坏真实感，降低决策深度）

### P1-1: `resolveMatch()` 是全知上帝函数

**问题**: 比赛引擎对双方队伍的所有数据拥有完全访问权，没有信息不对称。

**证据**（来自 `src/simulation/cup.ts` 第 74-123 行）:

```typescript
function resolveMatch(teamA, teamB, seed, round, detailed) {
  // ...
  const aPower = teamPower(teamA) + (scoreA >= 2 ? 3 : 0) + rng.int(-9, 9);
  const bPower = teamPower(teamB) + (scoreB >= 2 ? 3 : 0) + rng.int(-9, 9);
  // ...
  const star = bestPlayer(aWins ? teamA : teamB);
  eventCards.push({
    text: `${star.name} 打出关键影响...`,
  });
}
```

**为什么是问题**:

`teamPower()` 直接读取了对手的四维数值（firepower, tacticalExecution, cohesion, discipline）。在真实的经理游戏中，这些对手数据应该是**模糊的、需要侦察才能部分获得的**。

`bestPlayer()` 更是直接找出了对手的最强选手。这是赛前分析师花三天才能做出的判断，在你的代码里是一个 5 行的 sort 函数。

**修复预期**:
- 对手的四维数值在赛中应该是**隐藏的**或**通过侦察获得估计值**（允许偏差 ±5~10）
- 对手的"最强选手"应该在赛前侦察报告中揭示，而不是赛中上帝视角直接告诉你
- 赛中事件提到对手选手时，应该基于场上可观察的表现（"对方有一个选手位置感特别好"），而不是内部数据（"对方的 firepower 是 97"）

---

### P1-2: 赛前情报系统与赛中事件系统完全割裂

**问题**: 两个系统各自运行，互不影响。赛前侦察的选择不影响赛中事件的内容。

**证据**:

玩家在 S04（赛前事件卡）中做了侦察选择——比如选了"看 demo，拆他们的节奏"——`scout-briefings.ts` 的 `generateScoutCard()` 返回了一个 `ScoutChoiceFeedback`。但这个反馈的 `effect`（如 `{ tactics: 5, tactical_control: 4, firepower: -2 }`）只是数值修正，**没有改变赛中事件卡的内容和指向**。

赛中事件仍然可能随机到：
- "对手习惯快攻 B"——即使你没做侦察，也能看到这条
- "sh1ro 喜欢主动找首杀"——即使你侦察的是 ZywOo

**修复预期**:
- 赛前侦察选择的对手习惯信息 → 标记到比赛上下文 → 赛中事件卡有条件地展示相关文本
- 赛前选了侦察对手 AWP 选手 → 赛中可能出现"根据侦察报告，你们预判了 sh1ro 的架枪位"（有侦察才有这条）
- 赛前没有侦察 → 赛中你只能看到**通用战术描述**，看不到对手个人习惯

---

### P1-3: 对手经济状态完全透明

**问题**: 当前设计中，玩家总是知道对手的经济等级。

**证据**（设计意图，尚未完全实现在代码中，但在 `match-beats.ts` 和 `mechanic-events.ts` 的事件叙事中隐含了这一点）:

比赛中直接告知"对手经济状态全买"、"对手装备不足"等。

**为什么是问题**:

在真实 CS 中，你**推测**对手经济的方式是：
1. 看对方上回合买了什么装备（但被烟雾挡住就看不清）
2. 看对方是否 ECO（明显的手枪声）
3. 看对方是否 full buy（满道具）

但这些都不是 100% 确定的。你不知道对方会不会强起，你不知道对方是不是在演。

**修复预期**:
- 对手经济状态应该用**模糊语言**表达，而不是精确数据
- 例：不是说"对手 ECO 局"，而是说"对方这把脚步声很轻，听不到太多步枪"
- 如果你是 CT 方且对方 ECO，你们可能注意到"这把手枪声多"
- 赛前侦察可以提升你对对手经济的推断精度，但不应该直接给你精确答案

---

### P1-4: "全局解说员"叙事视角不合逻辑

**问题**: 事件卡采用了"上帝视角解说员"的口吻，描写了双方的心理活动。

**证据**（来自 `src/content/match-beats.ts`）:

```
第 731-734 行:
"场馆里一万两千人——刚才还在喊——突然安静了。那种安静不是没声音，是所有人同时屏住呼吸的安静。你能听到空调的嗡嗡声，能听到 {player} 鼠标的点击声，能听到他椅子的吱呀声。"

第 742 行:
"他的瞳孔在收缩。"

第 770 行:
"{player} 换了弹——满弹匣。但他没动。他在等。\n\n等什么？等对手犯错误。"
```

**为什么是问题**:

这些描写虽然文学性强，但从"经理人视角"来看毫无意义。你作为教练，坐在隔音棚外面，你看不到选手的瞳孔收缩，你不应该能听到对手的鼠标点击声。

更关键的问题：当你同时描写双方的心理活动时，你是在暗示玩家拥有读心术。

**修复预期**:
- 叙事视角必须锁定在**玩家队伍一侧**（教练视角+己方语音）
- 对手的行为只能通过**外部可观察的表现**来描述
- 可以保留"场馆气氛"描写（那是公共空间），但不能进入对手的心理或隔音棚内部
- 己方选手的心理可以描写——因为你是他们的教练，你知道他们的性格

## P2: 中度问题（积累后会严重损害体验）

### P2-1: 七个设计文档写了等于白写

以下设计文档中的系统在当前代码中的**通过率是 0%**：

| 设计文档 | 声明的系统 | 代码实现 |
|---|---|---|
| `MECHANICS.md` § Strategy Memory | 双方队伍的战术记忆追踪 | 不存在 |
| `MECHANICS.md` § Scouting Cards | 赛前情报转化成赛中优势 | 数值修正但不改变赛中叙事 |
| `MECHANICS.md` § Attack/Defense Matrix | 策略碰撞表 | 不存在（`resolveMatch` 是比较 teamPower） |
| `MECHANICS.md` § Timeout Decisions | 三选项暂停 | `mechanic-events.ts` 有事件但 `cup.ts` 没有调用 |
| `MECHANICS.md` § Coach Philosophy | 教练哲学影响 | `mechanic-events.ts` 有哲学选择事件但没有任何后续影响 |
| `MECHANICS.md` § Team Chemistry | 化学反应规则 | 球员数据有 traits 但 `cup.ts` 的 `resolveMatch` 完全不读取 |
| `GAMEPLAY.md` § 五个回合阶段 | opening-pressure-swing-adjustment-closing | `resolveMatch` 只是一个 for 循环 |

这不是"设计超前于实现"——这是**设计和实现活在两个平行宇宙**。

### P2-2: `bestPlayer()` 排序逻辑和数据内容矛盾

**证据**（来自 `src/simulation/cup.ts` 第 66-72 行）:

```typescript
function bestPlayer(team: TeamRoster): Player {
  return [...team.starters].sort((left, right) => {
    const leftScore = left.firepower + left.clutch + left.tactics * 0.35;
    const rightScore = right.firepower + right.clutch + right.tactics * 0.35;
    return rightScore - leftScore;
  })[0];
}
```

这个公式忽略了 `discipline`（纪律）。在 CS 中，纪律决定了选手会不会在关键时刻犯低级错误。但你的"最佳选手"算法认为纪律不重要。

这意味着一个 firepower 99, discipline 10 的"玻璃大炮"会被排在 firepower 90, discipline 85 的"定海神针"前面。这在数据层面可能是对的，但**不符合任何真实电竞分析逻辑**。

## P3: 细节问题（单独看不大，但多了就很恶心）

### P3-1: 事件库中"教练看到对手选手表情"是心灵感应

`mechanic-events.ts` 第 347 行：

```
"{star_player_name}握紧了鼠标。故事这东西，有时候比战术管用。"
```

第 444 行：

```
"{rival_team} 的指挥透过玻璃看到了。他的脸色——你想用'难看'形容，但难看都算好听的。"
```

**问题**: 你坐在自己的隔音棚/教练席，隔着两排电脑和一面隔音玻璃，你能看到对面指挥"脸色的微妙变化"？你是 8K 望远镜成精吗？

### P3-2: 赛前情报的"放烟雾弹"效果自相矛盾

`scout-briefings.ts` 中，"放烟雾弹"（bluff）选项的效果是 `{ tactics: 5, momentum: 2, discipline: -3 }`。

**问题**: 在社交媒体上泄露假情报**增加战术值**？这不合理。职业选手怎么可能因为在网上发了一条假消息就变得战术执行力更强？真正的效果应该是：下场比赛时，对手的防守可能会出现漏洞（因为情报被误导了）。但这不是"tactics +5"能表达的——这是需要引擎层面实现的"对手策略误判"。

### P3-3: $800 不够买的东西选项里却有

`scout-briefings.ts` 虽然不直接涉及经济，但整体叙事中从未提到过"手枪局的装备限制"这个事实。

## 完整的需要修复项清单（供另一个 agent 执行）

### 紧急（P0，必须立即修复才能算一个合格的游戏）

1. **删除所有赛中透视对手习惯的文本**
   - 搜索 `src/content/match-beats.ts` 和 `src/content/events.ts` 中所有直接描述对手战术偏好、个人习惯的内容
   - 重构为"可观察的行为"（"对方这把打得很激进" ← 允许，"对方喜欢主动找首杀" ← 禁止）

2. **删除所有引用对手语音的叙事**
   - 搜索 `"rival_team" + "在语音"` 和 `"说："` 等模式
   - 己方语音保留，对手语音全部删除

3. **重写 `opening_economy_choice` 为手枪局独有选项**
   - 文件：`src/content/mechanic-events.ts`
   - 手枪局选项：全甲+USP/Glock / 半甲+道具 / 沙鹰豪赌 / 省钱 / CT 拆弹套件

4. **建立赛前情报 → 赛中文本的桥接**
   - 如果赛前做了侦察，标记一个 `hasScouting` flag
   - 赛中事件卡根据 flag 切换文本："根据侦察报告，你们预判了……" vs 通用描述

5. **锁定叙事视角为教练视角**
   - 删除"对手的瞳孔收缩"、"对手的脸色变化"等上帝视角描写
   - 删除"全场一万二千人屏住呼吸"类的全知描写（除非是广播/公共信息）

### 重要（P1，应该在 v0.1 上线前完成）

6. **`resolveMatch` 中对手数据需要"战争迷雾"层**
   - 对手的四维数值应该给估计值（侦察做得越好，估计值越精确）
   - `bestPlayer()` 不应该在赛中直接返回对手最强选手

7. **对手经济状态使用模糊语言**
   - 不是"对手 full buy"，而是"对方装备声听起来很齐"
   - 侦察等级影响推断精度

8. **实现赛前侦察对赛中事件卡内容的实际影响**
   - 现在只有数值修正，需要改变文本指向

### 改进（P2-P3，可以在后续迭代中处理）

9. 给策略碰撞表、化学反应、教练哲学、战术记忆等系统写实际的引擎代码
10. 重写 `bestPlayer()` 包含 discipline 权重
11. 清理事件库中所有越界视角的描写

## 结论

这个游戏的核心问题是：**它假装是一个需要策略和准备的经理游戏，但实际上给了玩家一套完整的作弊工具。** 赛前侦察系统是装饰品，赛中信息是完全透明的，对手语音是公放的，选手习惯是直接标注的。

要修这个，不需要推翻重来。需要的是一刀一刀切掉这些"透视挂"，让玩家真正需要**通过赛前准备获取信息优势**，**通过赛中观察做出判断**，而不是等着游戏把答案喂到嘴边。

修完之后，`src/content/scout-briefings.ts` 里那 827 行精心设计的赛前情报系统才会真正有意义——因为玩家会知道：如果不在赛前做功课，比赛里就真的什么都不知道。
