# Office Hours: Game Mechanics

Generated by gstack office-hours on 2026-06-25.

Status: In progress

Mode: Builder with future commercial potential.

## Goal

The game is primarily a fun personal side project: make a lightweight CS esports manager that the creator genuinely wants to play. It should also preserve the option to become a small commercial or shareable game later.

## Captured Answers

| Question | Answer | Product Implication |
|---|---|---|
| Primary goal | A and E: fun side project plus future commercial potential | Optimize first for delight and a playable loop, but keep mechanics legible, replayable, and easy to explain to players. |
| Core delight | A, C, and D: dream-team roster building, text-story drama, and long-term three-year management | The game should prioritize budget-constrained star drafting, memorable narrative events, and season-to-season roster progression over deep round-by-round tactical realism. |
| Match granularity | Three compressed rounds/nodes per match. Opening node always gives an economy choice: full buy for immediate strength or eco/save to preserve later resources. Tactical choices include rush site, default/contact play, and other CS-like plans. Middle and late nodes can trigger event prompts such as real defuse versus fake defuse, continue holding angles, send a lurker, call timeout to break enemy rhythm, or recover morale. In single-player, the player chooses and AI auto-chooses the opponent response; in two-player, both sides can choose. | Each match should be short but choice-dense. The engine should model three major decision windows, strategy collision, resource preservation, morale, damage/losses, and momentum. It should not simulate every CS round. |
| Event philosophy | Events should happen before cups or individual matches: coach visa problems, players forced to call tactics themselves, great form, bad form, absence requiring a substitute, and similar reality-inspired disruptions. | The season needs a pre-match/pre-cup event system that changes control, stats, available players, morale, or narrative stakes. |
| Career ending | After three years, the game should produce a chronicle recording the whole run. | Endgame is not only a title count; it should generate a timeline of championships, collapses, absences, star moments, and defining decisions. |
| Player control during events | B: medium loss of control. Some events may remove player control for one match node, but never without a concrete story reason. The game must explicitly tell the player why control is unavailable at that moment. | Loss of control is allowed only when backed by narrative cause, visible effect, and preferably a recovery path. Random silent lockouts are forbidden. |
| Core resources | B: four-resource model. In-match resources include economy, team morale effects, player condition, and tactical control. Morale should not necessarily be a visible numeric value; morale actions can temporarily raise team attributes. | The engine should track enough state to make buy/save, timeout, condition, and tactical disruption meaningful without turning the UI into a spreadsheet. |
| In-match economy | A match uses a best-of-five compressed-round structure: first to three rounds wins. Economy follows CS-like streak logic: winning two rounds means the team still has money even after losing the third; losing the first means the second round is poor; losing two rounds means the third round has money again. When a team has no money, it can only choose an eco/save round, and poor equipment reduces firepower or related attributes. | Match simulation needs a local economy state that gates available choices and modifies player/team attributes. |
| Player interaction cadence | Opening tactical choice is mandatory for each match. Additional prompts should appear occasionally through events, clutches, timeouts, economy crises, or turning points. Frequency should create participation without dragging the pace. | The game should not ask for a choice every compressed round by default. It should guarantee one opening choice and then use event-driven prompts. |
| Opening decision structure | A: opening choice is split into two steps: economy choice and tactical choice. Economy covers full buy/partial buy/eco when available. Tactics cover plans such as rushing a bombsite, default/contact play, slow lurk, or fake/rotate. | The first implementation should not collapse economy and tactics into one combined button; the player should understand buy strength and tactical plan as separate levers. |
| Tactical execution | Player tactics, team cohesion/chemistry, discipline, and individual player stats determine how well a chosen tactic executes. If both sides choose similar or clashing tactics, cohesion/discipline differences can decide who loses more, but elite individual stats can compensate for or amplify the mismatch. | The strategy engine needs team-level stats beyond total firepower: at minimum cohesion/chemistry and discipline/tactical execution. Tactical mirrors should be resolved by execution quality, not pure randomness. |
| Mode scope | The game now only considers single-player mode. The player faces AI-controlled opponents. Non-player bracket matches are not fully simulated; they quickly pick/randomize an advancing team into later rounds such as semifinals or finals. | Remove local/two-player requirements from v0.1. Invest detail into player-facing matches and keep offscreen AI matches fast. |
| Team attributes | B: teams use four 1-100 attributes: total firepower, tactical execution, cohesion/chemistry, and discipline. Players should be able to see these values after decisions resolve. | Team stats should be transparent enough for players to understand why a tactic worked, failed, or partially succeeded. Decision result UI should show stat changes or relevant checks. |
| Decision feedback UI | Match text appears as a vertical sequence of event cards/boxes. Each box is a random or scripted match event. Some boxes contain player decisions. After the player chooses, that box continues downward with the resulting battle description. The final line of the box shows numeric stat changes such as "team cohesion +4" or similar. | The UI should use event cards as the basic match log unit. Decision feedback should be narrative first, with a compact numeric delta at the bottom of the card. |
| Event density | B: medium rhythm. A player-facing match should usually contain 7-10 event cards, with 2-3 player decision cards. | This should provide enough story and participation without making a cup feel slow. |
| Feedback transparency | Non-decision event cards use semi-transparent explanation: labels plus a few key values, such as "your tactical execution was better, but equipment was weaker." Player decision cards use more transparent quantified feedback after resolution, including relevant stat comparison and final deltas. | Random events should preserve story flow. Player decisions should make cause and effect clearer so choices feel learnable and fair. |
| Quantified decision detail | B: decision cards show team-level comparison plus key player influence. Example: tactical execution 82 vs 74, ZywOo clutch +8, apEX personality causes discipline -3, team cohesion +3. | The game should surface stars and traits in decisions without showing full formulas by default. |
| Personality traits | B: traits provide stable stat modifiers and can trigger occasional event chains. Example: a hot-blooded player can hurt team cohesion, force an aggressive push, spend too long lurking, cause early casualties, or lose a bombsite and retreat. | Traits should create both balance texture and memorable story arcs. Event chains must remain explainable and tied to the trait. |
| Timeout intervention | A match has one timeout opportunity. The game should offer the timeout prompt during crisis or momentum events: a volatile player trait triggers, the team falls into poor form, the opponent enters hot form, or the opponent builds a win streak. | Timeout is the player's limited emergency brake. It should interrupt negative chains, restore some tactical control or morale effect, and create an important choice about whether to spend the only timeout now. |
| Cohesion protection | High team cohesion can reduce the impact of low morale, poor individual form, anger, and internal conflict events. | Cohesion should be a defensive team-building stat, not only a tactical mirror-breaker. It makes stable teams less likely to collapse when personalities or pressure flare up. |
| Cohesion calculation | D: cohesion should consider each player's individual personality score, personality/tag combinations, and real-world teammate familiarity bonuses. | Team cohesion should reward thoughtful roster construction. A dream team can still suffer if personalities clash, while former teammates can gain a small stability bonus. |
| Team shape | Each team has six people: five active players and one substitute. Individual players have firepower, tactics, and personality. Teams also need aggregate stats such as total firepower and harmony, though the exact team-stat model is still open. | Roster content and draft rules must support substitutes from the beginning or by the first season-mode milestone. |
| Out-of-match economy | Outside matches, the coach/manager uses budget to buy players and configure the roster. Between cups, players can be traded using money and possibly player exchange. Cup placement pays prize money; better results increase future roster freedom, while poor results limit access to stars. | Season mode needs prize tables, transfer windows between cups, player prices, and roster validation. |
| Initial roster and transfers | At game start, the player uses starting money to select six players from the player pool. After that, roster changes require trades with other teams, using players and money as exchange chips. | The game should not allow free-market reshuffling after the initial draft. Transfer UI must support player-plus-cash trade offers. |
| Team pool structure | The active tournament has the player's custom team plus seven AI teams. The player's team name is "枪神队伍". The player drafts from the eight-team player pool; remaining players are then assigned to their source teams when possible, and missing slots are filled randomly. | The game world should become seven AI teams plus the player's new organization after draft, while still using the curated eight-team roster pool as source material. |
| Trade acceptance | A plus C: start with simple value matching, but include player willingness. Trades compare money plus outgoing player value against target value, and also consider whether the player is willing to join based on factors such as team success, role, cohesion, and possibly personality. | The first trade system should be understandable but not purely mechanical. Player willingness adds story without requiring full AI GM complexity. |
| Budget and pricing | Starting budget can use an abstract value such as 100. Player strength should be designed from real-world-inspired ability first, then price should be derived from those ratings. | Balance starts with believable player ratings, then converts them into prices so stars feel expensive for gameplay reasons. |
| Elimination flow | If 枪神队伍 is eliminated from a cup, the player does not watch the rest of the bracket. The game quickly announces the cup champion and moves to the next cup phase. | Keep pacing fast. Detailed match simulation is only for the player's team. |
| Player team identity | 枪神队伍 is a Chinese team. | Events can use Chinese fan expectations, regional pride, travel pressure, public opinion, and sponsor pressure as story hooks. |
| Real names | Prototype uses real player names directly. | Roster content can use names such as donk, ZywOo, NiKo, and apEX, but negative/private-life events must stay generic and non-defamatory. |
| First playable version | v0.1 is accepted when the player can complete one cup. | Do not wait for the full one-year or three-year campaign before building a playable slice. |
| Annual top players | At the end of each year, the game announces the season's top ten players across all eight teams, not only the player's team. Ratings can be inspired by HLTV-like performance logic. | The season summary needs a global player ranking system based on simulated performance, star moments, and stats. This supports long-term story and market value. |
| Cup and annual awards | Each cup announces champion and cup MVP. Each year announces best club and top ten players. | Awards should convert simulation outputs into memorable history, market value, and the final three-year chronicle. |
| Event scope | Events can trigger in-match, between cups, and at different tournament stages. The event library should be large. Choices inside events can increase or reduce different effects. | Build the event system as a reusable content-driven library, not only as match log flavor. Events should be tagged by timing, severity, affected stats, and choice outcomes. |
| Non-match screen | When not in a match, the central screen should show the eight-team bracket/tree schedule. | The primary hub should be the tournament bracket, not a generic menu. |
