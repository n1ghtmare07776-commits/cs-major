# Final Three-Year Mode Implementation Slices

> **给主线程/接手代理的说明：** 这是贴近当前真实进度的分段实现文档，不是假定项目已经完成的“理想计划”。目标是把现有“第一杯可玩原型”稳定扩展为“三年完整可通关版本”。

## 终极验收标准

这一版只有在以下三条同时满足时，才算完成：

1. **美工与对话文本全都载入**
   - 已有 UI 风格、杯赛视觉 token、事件文本包、比赛内普通战况框、杯间事件、赛后鼓励/安慰文本都实际接入运行时，而不是只存在文档里。
2. **玩家能完整完成三年的杯赛**
   - 从选人建队开始，连续打完 `3 年 x 3 杯 = 9 杯`，中间没有软锁、空白页、按钮失效、状态丢失。
3. **功能全部齐全**
   - 包含：6 人阵容、BO5 比赛、每场约 10 个事件框且玩家决策约 3 个（上限 5 个）、动态树形对阵图、杯赛/年度奖项、杯间事件与轻量交易、三年编年史、打包成可运行 app。

---

## 当前真实进度

### 已完成

- 第一阶段可玩原型已经存在，能从组队一路推进到第一杯的完整黄金路径。
- 当前浏览器版本已有：
  - `6` 人阵容选择（`5` 首发 + `1` 替补）
  - 树形对阵图基础 UI
  - 首场玩家比赛扩展到约 `10` 张事件卡
  - 决策卡控制在约 `3` 张
  - 首发五人的 `K/D/A/Impact` 展示
  - 选手阵亡后状态框变暗/灰化
- 计划文档、QA 文档、美术/内容映射文档都已经补齐。
- 现有基础测试和构建命令可运行，第一阶段已被用户人工验收过。

### 未完成

- 还没有真正的 **三年九杯赛季状态机**。
- 还没有完整的 **杯间循环 / 轻量交易 / 年度奖项 / 三年编年史**。
- 对阵图还没有完全根据三年赛程动态推进。
- 文本包和美工要求还没有“全量载入”，目前只是局部接入。
- 数值平衡还不够，`donk + ZywOo` 这类双明星开局仍然偏强。
- 还没有完成最终 **app 打包** 与完整全流程验收。

## 分段 1：先稳住现有原型与内容入口

**目标**

- 保住现有第一杯可玩路径，不让后续重构把已验收内容打坏。
- 盘清“哪些美工/文本资源已经在代码里接入，哪些还只是文档要求”。
- 为后三年模式建立统一内容入口，避免继续把逻辑和文案都堆在 `browser.js` 里。

**涉及文件**

- `src/app/browser.js`
- `src/content/events.ts`
- `src/content/mechanic-events.ts`
- `src/content/match-beats.ts`（若不存在则创建）
- `src/content/awardsText.ts`（若不存在则创建）
- `src/content/epilogue.ts`（若不存在则创建）
- `src/ui/styles.css`
- `tests/app/browser-flow.test.mjs`

**先后依赖**

- 这是所有后续开发的地基，应当最先做。
- 不依赖三年状态机，但会为其提供稳定的内容接入层。

**验收点**

- 第一杯黄金路径回归通过。
- 已有比赛事件、KDA、阵亡灰化、树形图不回退。
- 能明确区分：
  - 已接入的文本
  - 待补写的普通战况框
  - 待接入的杯间/年度/编年史文本

**容易翻车的地方**

- 一边拆内容入口，一边把当前已验收按钮流程弄断。
- 把文档里的文本包路径理解错，导致运行时找不到内容。
- 继续把新逻辑硬塞进 `browser.js`，后面会越来越难收口。

## 分段 2：做三年九杯的赛季状态机

**目标**

- 让项目第一次具备真正的 `3 年 x 3 杯 = 9 杯` 主循环。
- 明确每个阶段的状态切换：组队、杯赛、赛后、杯间、年度奖项、编年史。

**涉及文件**

- `src/domain/campaign.ts`
- `src/domain/awards.ts`
- `src/simulation/season.ts`
- `src/simulation/cup.ts`
- `src/content/cups.ts`
- `src/app/browser.js`
- `src/app/game.ts`
- `tests/simulation/season.test.ts`
- `tests/app/three-year-flow.test.mjs`

**先后依赖**

- 依赖分段 1 的内容入口整理。
- 先于 UI 大改、年度奖项、打包。

**验收点**

- 固定 seed 下可以完整跑完 9 杯。
- 玩家被淘汰后仍能进入杯赛总结、杯间阶段、下一杯。
- 第 1 年、第 2 年结束后会进入年度奖项并进入下一年。
- 第 3 年结束后会进入最终编年史，而不是回到开局。

**容易翻车的地方**

- 状态太多却没有统一模型，最后全靠 `if/else` 拼接。
- 玩家比赛与 AI 场次推进规则混在一起，导致某些杯赛卡死。
- 三年状态推进写通了，但没有把“当前杯赛是哪一站、当前是第几年”正确显示给 UI。

## 分段 3：扩成完整 BO5 比赛引擎

**目标**

- 从“第一场原型对局”升级到可复用的 BO5 比赛引擎。
- 保证每场玩家比赛都符合目标节奏：
  - 大约 `10` 个事件框
  - 大约 `3` 个玩家决策
  - 上限 `5` 个决策
  - 其余为普通战况叙事框

**涉及文件**

- `src/simulation/match.ts`
- `src/simulation/cup.ts`
- `src/content/match-beats.ts`
- `src/content/mechanic-events.ts`
- `src/content/events.ts`
- `src/app/browser.js`
- `tests/simulation/match.test.ts`
- `tests/app/three-year-flow.test.mjs`

**先后依赖**

- 依赖分段 2 的赛季状态机。
- 先于数值平衡和大规模文本润色。

**验收点**

- 玩家比赛为 `BO5`，先赢 `3` 图获胜。
- 每场比赛都能生成 `10-15` 个框，目标表现为“约 10 个”。
- 决策卡不超过 `5`，通常在 `3` 个左右。
- 普通战况框能稳定出现 CS 风格叙述：
  - 谁击杀了谁
  - 双杀/三杀
  - 补枪、回防、保枪、假打、反清、强起、ECO 等
- 比赛中 KDA、阵亡灰化、比分带、事件框联动正常。

**容易翻车的地方**

- 卡片数达标了，但大半都是空洞 filler 文本，玩起来像凑数。
- 决策卡控制不住，后面一改就超过 5 个。
- BO5 只改了比分显示，实际仍然按一局逻辑在跑。

## 分段 4：做数值平衡与化学反应

**目标**

- 解决“买两个超巨就能无脑通关”的问题。
- 让队伍契合、纪律、战术、替补、事件抗性真正参与胜负。

**涉及文件**

- `src/content/balance.ts`
- `src/content/players.ts`
- `src/content/traits.ts`
- `src/content/teams.ts`
- `src/simulation/match.ts`
- `src/simulation/cup.ts`
- `src/simulation/season.ts`
- `tests/simulation/balance.test.ts`

**先后依赖**

- 依赖分段 3 的比赛引擎定型。
- 先于最终 QA 和最终打包。

**验收点**

- 双明星阵容依然能赢，但不能在多 seed 扫描中形成明显碾压。
- 中价位、纪律高、体系好的阵容有真实争冠可能。
- 替补不只是装饰，会在事件与模拟结果中体现价值。
- 奖项与 MVP 不是单纯按火力值排序。

**容易翻车的地方**

- 只改单个明星价格，结果仍然被玩家轻松买穿。
- 化学反应写了很多规则，但比赛里几乎感知不到。
- 平衡只看一条黄金路径，不跑多 seed，最后上线仍然崩。

## 分段 5：完成三年模式的 UI、美工与交互

**目标**

- 把当前单杯原型 UI 扩展成真正的“三年控制室”。
- 杯赛之间以树形对阵图为中心，比赛时是事件框战斗室，年度与结局有独立展示面。

**涉及文件**

- `src/ui/csCupManagerUi.ts`
- `src/ui/styles.css`
- `src/app/browser.js`
- `src/app/game.ts`
- `docs/design-docs/final-three-year-mode-art-and-content-map.md`
- `tests/app/three-year-flow.test.mjs`

**先后依赖**

- 依赖分段 2 的状态机、分段 3 的比赛框架。
- 可与分段 4 后半段并行收尾，但不要先于状态模型大改。

**验收点**

- 玩家在任意时刻都能一眼看懂：
  - 第几年
  - 当前杯赛
  - 当前阶段（对阵图 / 赛前 / 比赛 / 奖项 / 杯间）
- 对阵图根据真实模拟结果推进，不是静态占位。
- 组队和换人交互比现在更顺滑，选取/取消选择清晰。
- 比赛室不是简单弹窗，而是完整战斗界面：
  - 左侧/中部事件流
  - 右侧决策区
  - 右下或侧边 KDA 面板
  - 阵亡灰化联动
- 年度奖项与最终编年史页有独立视觉层级，不像 debug 面板。

**容易翻车的地方**

- UI 先行重做，结果后面的状态和内容接不上。
- 赛程树做得好看，但不能承载真实数据推进。
- 交互改顺手时把“已选 6 人、首发/替补标识”改乱。

## 分段 6：把文本包与事件库全量接入

**目标**

- 把现有文档和文件夹中的对话文本、普通战况框、杯间事件、赛后鼓励/安慰文本真正接入运行时。
- 统一成贴近 CS 术语的中文表达。

**涉及文件**

- `src/content/events.ts`
- `src/content/mechanic-events.ts`
- `src/content/match-beats.ts`
- `src/content/awardsText.ts`
- `src/content/epilogue.ts`
- `src/content/cupFlavor.ts`（若不存在则创建）
- `src/app/browser.js`
- `docs/design-docs/final-three-year-mode-art-and-content-map.md`

**先后依赖**

- 依赖分段 3 的比赛卡片结构、分段 5 的 UI 展示位。
- 先于最终打包和最终 QA。

**验收点**

- 比赛内普通事件框数量足够，不再靠少数大事件硬撑整局。
- 杯赛结束会宣布冠军与杯赛 MVP，并有合适的鼓励/安慰文本。
- 每年结束会宣布最佳俱乐部、年度前十选手，并有赛季总结文本。
- 三年结束会生成编年史，并插入鼓励/安慰/回顾文案。
- 全局术语统一为玩家熟悉的 CS 表达，不出现明显错别字和奇怪翻译。

**容易翻车的地方**

- 文本很多，但没有按“赛前 / 局内 / 杯间 / 年度 / 结局”分发到正确时机。
- 术语没统一，同一概念在不同页面说法不一样。
- 文本全部载入了，但节奏过满，导致轻量化目标被破坏。

## 分段 7：打包、全流程验收与交接

**目标**

- 让项目不只是本地 dev 原型，而是能交付一个可运行 app。
- 做一次真正面向“完整三年流程”的终验，而不是只看第一杯。

**涉及文件**

- `package.json`
- `electron/` 或当前桌面打包入口目录（按现有项目实际创建）
- `src/app/main.ts`
- `tests/app/three-year-flow.test.mjs`
- `tests/simulation/season.test.ts`
- `tests/simulation/match.test.ts`
- `tests/simulation/balance.test.ts`
- `docs/QA-final-three-year-mode.md`
- `docs/HANDOFF_PROGRESS.md`

**先后依赖**

- 必须放在最后。
- 依赖前 1-6 段全部达到“可串联”状态。

**验收点**

- `npm run build`
- `npm run test`
- `npm run check:docs`
- `npm run test:architecture`
- 打包命令通过，生成可运行产物。
- 打包后的 app 能启动、开始新游戏、打完至少一整杯，且三年完整流程本地可手动走通。
- 更新交接文档，明确最终已完成项、已知限制、运行方式。

**容易翻车的地方**

- Web 版能跑，打包后路径或资源加载挂掉。
- 只测赢线不测输线，导致某些被淘汰分支软锁。
- 以“文档写完了”代替“文本和美工真的载入运行了”。

## 主线程建议的执行顺序

### 最先做的前两段

1. **分段 1：先稳住现有原型与内容入口**
2. **分段 2：做三年九杯的赛季状态机**

原因很直接：

- 现在最大的风险不是“缺一个页面”，而是主循环还没真正建立。
- 如果不先把内容入口和状态机收好，后面做比赛、美工、文本接入都会反复返工。

## 阶段标题总览

1. 先稳住现有原型与内容入口
2. 做三年九杯的赛季状态机
3. 扩成完整 BO5 比赛引擎
4. 做数值平衡与化学反应
5. 完成三年模式的 UI、美工与交互
6. 把文本包与事件库全量接入
7. 打包、全流程验收与交接
